# Changelog Specification

The governance layer is code — it should be versioned like code. The changelog tracks:
1. **What changed** in the rules, patterns, and manifest
2. **Why it changed** — driven by task outcomes, QC findings, or /evolve reports
3. **When it changed** — timestamped entries
4. **What version** — semantic versioning of the governance layer itself

## Versioning Scheme

The governance layer uses semantic versioning: `MAJOR.MINOR.PATCH`

| Bump | When | Example |
|------|------|---------|
| **MAJOR** | Breaking changes: restructured manifest, removed rules, changed agent behavior fundamentally | Rules reorganized, manifest schema v2 |
| **MINOR** | New capabilities: new rules, new patterns, new modules in boundaries, new security checks | Added billing module boundaries, new GDPR check |
| **PATCH** | Refinements: updated pattern examples, tweaked rule wording, added translations to conventions | Updated controller pattern with new middleware style |

### Version Storage

The current version lives in `manifest.json` under a `governance` field:

```json
{
  "governance": {
    "version": "1.2.0",
    "initialized": "2026-03-17",
    "last_evolved": "2026-03-25",
    "changelog": ".claude/CHANGELOG.md"
  }
}
```

## Changelog Format

Follow [Keep a Changelog](https://keepachangelog.com/) conventions:

```markdown
# Changelog

All notable changes to this project's governance layer are documented here.

Format follows [Keep a Changelog](https://keepachangelog.com/).
Versioning follows [Semantic Versioning](https://semver.org/).

## [Unreleased]

### Added
- New security check for GDPR field detection

## [1.2.0] - 2026-03-25

### Added
- Billing module added to manifest boundaries
- New pattern file: `patterns/job.md` for queue jobs

### Changed
- Controller pattern updated: middleware now applied per-method
- Security rule: CSRF check now covers API routes

### Removed
- Deprecated `patterns/blade.md` — project fully moved to Inertia

### Fixed
- False positive: QC was flagging scoped queries as unscoped

## [1.1.0] - 2026-03-20

### Added
- ISMS module boundaries
- Translation convention: all 4 locales required

## [1.0.0] - 2026-03-17

### Added
- Initial governance layer generated by `/init`
- Manifest with stack, architecture, conventions, boundaries
- Security rules: auth bypass, tenant isolation, data exposure
- Architecture rules: module boundaries, layer enforcement
- Convention rules: naming, validation, translations
- Pattern files: controller, model, test, migration, component
- Agent skills: preflight, qc, evolve
```

## When to Update

### Automatic (agent-driven)

| Event | Action |
|-------|--------|
| `/init` completes | Create changelog with `[1.0.0]` entry |
| `/evolve` applies changes | Add entry under `[Unreleased]`, bump version |
| QC finds a false positive that leads to rule update | Add `### Fixed` entry |
| New pattern file created | Add `### Added` entry |

### Manual (user-driven)

Users may edit rules directly. When they do, they should add an `[Unreleased]` entry. The `/evolve` agent will assign a version number on next run.

## Integration with Other Skills

### /init
Creates the initial `[1.0.0]` entry listing everything generated.

### /evolve
After applying updates:
1. Move `[Unreleased]` items into a new versioned section
2. Determine version bump (major/minor/patch) based on change types
3. Update `governance.version` in manifest.json
4. Timestamp the new version

### /qc
QC does NOT write to the changelog directly. But if a QC finding leads to a rule change (via /evolve or manual edit), that change gets logged.

### Task Docs
Task closures don't directly update the changelog. But if a task's "lessons learned" cause a rule update, that update gets a changelog entry.

## Principles

- **One entry per logical change.** Don't split related updates into separate entries.
- **Why, not just what.** "Added GDPR check for new user_ip field in audit logs" not just "Added GDPR check."
- **Categories matter.** Use Added/Changed/Removed/Fixed consistently.
- **Version bumps are deliberate.** Don't auto-bump on every tiny change. Accumulate in `[Unreleased]` and bump when meaningful.
- **The changelog is for humans.** Write it so someone unfamiliar with the project can understand what changed and why.
